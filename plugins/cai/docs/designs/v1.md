## 1. Problem statement

AI 코딩 에이전트는 세션 간 기억이 없다. 프로젝트의 컨벤션, 아키텍처 결정, 알려진 문제, 미래 계획을 매번 다시 설명해야 하며, 설명이 빠지면 에이전트가 과거 실수를 반복하거나 프로젝트 방향에 반하는 코드를 생성한다. 단일 CLAUDE.md 파일은 소규모 프로젝트에서는 동작하지만, 50K+ LOC 규모에서는 한계에 도달한다.

본 설계는 Vasilopoulos의 3-tier codified context 아키텍처(arXiv:2602.20478)를 기반으로, Claude Code의 Skill + Agent + MCP 빌딩 블록에 맞게 재구성한 범용 도구를 정의한다.

## 2. Design goals

- **Seamless**: 개발자의 기존 Claude Code workflow에 마찰 없이 녹아들 것.
- **Incremental**: 처음부터 완벽할 필요 없이, 실패에서 점진적으로 성장할 것.
- **Tool-agnostic outputs**: 산출물(context 문서)은 plain markdown이어서 어떤 도구든 읽을 수 있을 것.
- **Ownership clarity**: tool이 관리하는 영역과 개발자가 소유하는 영역이 명확히 분리될 것.

## 3. Architecture overview — 3 building blocks

시스템은 세 가지 빌딩 블록으로 구성되며, 각각은 고유한 강점을 가진다.

**역할 분리 원칙:**
- Skill = 뼈대를 세운다 (구조 생성, 검증, 업그레이드).
- Agent = 살을 붙인다 (코드 분석 → 문서 생성, 리뷰).
- MCP = 접착제 (빠른 메타데이터 연산, 인덱싱).

### 3.1 Skills (commands)

사용자가 명시적으로 호출하는 절차적 명령.

| Skill | 역할 |
|-------|------|
| `cc-init` | 디렉토리 구조 생성, 빈 템플릿 배치, MCP 서버 등록 |
| `cc-onboard` | Onboarding pipeline orchestration (Phase 1-3) |
| `cc-drift-check` | spec ↔ code 정합성 검증 리포트 |
| `cc-add-spec` | spec 템플릿 생성 → spec-writer + context-interviewer 호출 |
| `cc-add-decision` | context-interviewer로 ADR 기록 |
| `cc-add-agent` | agent 템플릿 생성 → agents/에 파일 추가 |
| `cc-add-roadmap` | context-interviewer로 미래 계획 기록 |
| `cc-upgrade` | tool layer만 업데이트, project layer 보존 |

### 3.2 Agents (autonomous workers)

코드를 읽고 판단해서 산출물을 만드는 자율적 행위자. `.claude/agents/`에 markdown으로 정의.

**Onboarding pipeline agents:**

| Agent | 역할 | Phase |
|-------|------|-------|
| `structure-scanner` | 디렉토리/파일 분석, 모듈 경계 식별 | 1 (자동) |
| `module-analyst` | 단일 모듈 내부 구조 분석 (×N 병렬) | 1 (자동) |
| `verification-agent` | 생성된 spec의 claim을 소스 코드와 cross-validate | 1 (자동) |
| `context-interviewer` | 개발자 인터뷰로 project.md, decisions, roadmap 작성 | 2 (대화) |
| `relationship-mapper` | 모듈 간 의존 관계, 데이터 흐름 매핑 | 3 (자동) |
| `convention-extractor` | 반복 패턴 추출 (naming, error handling 등) | 3 (자동) |
| `constitution-assembler` | 산출물 종합 → rules, trigger table 구성 | 3 (자동) |

**Persistent agents:**

| Agent | 역할 | 호출 시점 |
|-------|------|----------|
| `context-gardener` | stale spec 업데이트, 새 패턴 발견 시 spec 제안 | post-change |
| `code-reviewer` | convention 위반 탐지, post-change 리뷰 | post-change |
| `spec-writer` | 소스 읽고 spec 작성, 기존 spec과 일관성 검증 | ad-hoc |
| `context-interviewer` | ADR/roadmap 등 사람 지식이 필요한 문서 작성 | ad-hoc |

### 3.3 MCP server (lightweight utility layer)

LLM 호출 없이 수행 가능한 빠른 메타데이터 연산. 서버 시작 시 `context/` 전체를 메모리에 로드한다 (100개 문서 ≈ 400KB, trivial).

**MCP tools:**

| Tool | 용도 | 입력 |
|------|------|------|
| `suggest_context(path)` | 파일 수정 전 관련 context 반환 | file path |
| `search_context(query, scope?, level?)` | 키워드로 문서 검색 | keywords + filters |
| `scan_drift()` | 전체 stale 체크 (git log vs frontmatter) | none |
| `suggest_agent(task)` | 적합한 agent 추천 | task description |
| `list_specs(filter?)` | 인벤토리 조회 | type, level, status |
| `dep_graph(scope?)` | 의존 관계 그래프 | module or spec path |

**In-memory indexing (3단계 검색 우선순위):**

```
tag_index:     frontmatter tags → [doc paths]           (수동 관리, 정확)
path_index:    src file path → [doc paths]              (covers 필드 역인덱스)
content_index: heading/identifier keywords → [doc paths] (자동 추출)
```

검색 순서: tag match → heading/identifier match → 본문 substring match.

**응답 크기 제약:** Claude Code MCP output 기본 한도 10K 토큰. 모든 tool response는 path + snippet(200자) + metadata만 반환. 전체 내용은 에이전트가 파일을 직접 읽는다.

**파일 변경 감지:** 서버 시작 시 전체 빌드. 런타임 중 파일 변경은 watcher로 해당 문서만 리로드 후 인덱스 리빌드 (100개 문서 기준 ms 단위).

## 4. Knowledge taxonomy

프로젝트 지식을 시제(temporal axis)로 분류한다. 각 type은 에이전트에게 서로 다른 행동을 요구한다.

### 4.1 Types and agent behavior

| 시제 | Type | 에이전트 행동 |
|------|------|--------------|
| 정체성 | `project` | 프로젝트의 목적과 제약을 이해하고 작업 |
| 과거 | `decision` (accepted) | 이 결정을 존중하라. 뒤집지 마라. |
| 과거 | `decision` (deprecated/superseded) | 더 이상 유효하지 않음. superseded_by 참조. |
| 현재 | `spec` | 이 패턴을 따르라. |
| 현재 | `convention` | 항상 이 규칙을 적용하라. |
| 현재 | `issue` | 이 문제를 악화시키지 마라. |
| 현재 | `glossary` | 이 용어를 이 의미로 사용하라. |
| 미래 | `roadmap` (planned/in-progress) | 이 방향을 의식하고 작업하라. 대규모 투자 자제. |
| 미래 | `roadmap` (exploring) | 참고만. 확정되지 않음. |

### 4.2 Directory structure

```
context/
├── project.md                          ← 프로젝트 정체성, 배경, 비전
├── glossary.md                         ← 도메인 용어 정의 (Ubiquitous Language)
│
├── decisions/                          ← ADR (시간순, flat, append-only)
│   ├── 001-chose-postgresql.md
│   ├── 002-event-driven-arch.md
│   └── 003-monorepo-structure.md
│
├── issues/                             ← Known issues / tech debt (flat)
│   ├── perf-n-plus-one.md
│   └── legacy-auth-migration.md
│
├── conventions/                        ← 코딩 컨벤션 / 패턴 (flat)
│   ├── error-handling.md
│   ├── testing-strategy.md
│   └── api-design.md
│
├── specs/                              ← 서브시스템 기술 스펙 (계층적)
│   ├── _overview.md                    ← 전체 아키텍처, 모듈 간 관계
│   ├── auth/
│   │   ├── _overview.md                ← auth 모듈 설계
│   │   ├── jwt-service.md              ← 컴포넌트 레벨
│   │   └── oauth-provider.md
│   ├── payment/
│   │   ├── _overview.md
│   │   ├── stripe-integration.md
│   │   └── refund-flow.md
│   └── notification/
│       ├── _overview.md
│       └── email-sender.md
│
└── roadmap/                            ← 미래 계획 (status별)
    ├── planned/
    │   └── migrate-to-oauth2.md
    ├── exploring/
    │   └── graphql-migration.md
    └── completed/
        └── initial-launch.md
```

`_overview.md`가 각 계층의 접착제 역할. Project-level overview는 모듈 간 관계를, module-level overview는 내부 컴포넌트 간 관계를 담는다.

**계층적 spec의 MCP 연동:** `suggest_context(path)`가 file path로 조회하면 component spec → parent module overview → project overview 순으로 zoom level을 제공한다. 에이전트는 작업 범위에 따라 필요한 깊이만 로드.

### 4.3 context/ 디렉토리명 충돌 대응

일부 프로젝트는 `context/`를 다른 용도로 사용할 수 있다 (Go context 패키지, React context 등). `codified-context.md` 설정에서 루트 디렉토리명을 변경 가능:

```markdown
<!-- .claude/rules/codified-context.md -->
Context directory: .context
```

기본값은 `context/`. 대안: `.context`, `docs/context`, `knowledge` 등.

## 5. Frontmatter schema

### 5.1 설계 원칙: 경량화

Frontmatter 필드가 많아지면 유지보수 부담이 커지고, 이는 논문의 핵심 실패 모드(stale spec)와 동일한 문제를 야기한다. **공통 필수 필드를 최소화**하고, type/level별 선택 필드를 분리한다. 자동 관리 가능한 필드는 tool/MCP가 관리.

### 5.2 Field definitions

**공통 필수 (모든 context 문서):**

| Field | Type | 설명 | 관리 주체 |
|-------|------|------|----------|
| `type` | enum | decision, issue, spec, convention, roadmap, glossary, project | 생성 시 설정 |
| `tags` | string[] | 키워드 태그 (MCP 검색의 1순위 인덱스) | 사람 |
| `last_commit` | string | 마지막 코드 동기화 commit hash | tool (자동) |

**공통 자동 관리 (tool이 채움, 사람이 안 건드림):**

| Field | Type | 설명 | 관리 주체 |
|-------|------|------|----------|
| `last_synced` | date | scan_drift가 마지막으로 확인한 날짜 | MCP (자동) |

**spec 전용:**

| Field | Type | 설명 | 관리 주체 |
|-------|------|------|----------|
| `level` | enum | project, module, component | 생성 시 설정 |
| `covers` | string[] | 이 spec이 다루는 소스 파일 glob 패턴 | 사람 (생성 시 자동 제안) |
| `parent` | string | 상위 overview 경로 | tool (자동) |
| `confidence` | enum | draft, reviewed, verified | 사람 |

**module overview 전용 (추가):**

| Field | Type | 설명 |
|-------|------|------|
| `components` | string[] | 하위 컴포넌트 목록 |
| `exports` | string[] | 다른 모듈이 의존하는 public interface |
| `depends_on` | string[] | 의존하는 다른 모듈/문서 |

**project overview 전용 (추가):**

| Field | Type | 설명 |
|-------|------|------|
| `modules` | string[] | 전체 모듈 목록 |
| `module_dependencies` | map | 모듈 간 의존 관계 |

**decision 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `status` | enum | proposed, accepted, deprecated, superseded |
| `superseded_by` | string | 대체 decision 경로 |

**roadmap 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `status` | enum | exploring, planned, in-progress, completed |
| `target` | string | 목표 시기 (e.g., 2026-Q3) |
| `related_specs` | string[] | 영향받을 수 있는 spec (soft link) |

**issue 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `severity` | enum | low, medium, high, critical |

### 5.3 주요 설계 결정

**Roadmap soft linking:** 초기 설계에서 `invalidates` 필드로 "이 계획이 실행되면 이 spec이 무효화 된다"는 strong claim을 했으나, 미래에 대한 예측이므로 misleading 위험. `related_specs`로 약하게 연결하고 soft warning만 제공.

**depends_on 사용 범위 제한:** 초기에는 spec의 module overview에서만 사용. 나머지 type 간 연관은 tags 기반 MCP 검색으로 충분. 필요성 입증 시 점진적 확대.

### 5.4 예시

```yaml
# specs/auth/_overview.md
---
type: spec
level: module
tags: [auth, security, jwt]
covers: [src/auth/**, src/middleware/jwt.ts]
last_commit: a3f7b2c
last_synced: 2026-04-02
confidence: reviewed
components: [jwt-service, oauth-provider]
exports: [AuthService.validateToken(), AuthService.createSession()]
depends_on: [context/specs/payment/_overview.md]
---
```

```yaml
# roadmap/planned/migrate-to-oauth2.md
---
type: roadmap
status: planned
tags: [auth, security, migration]
target: 2026-Q3
related_specs: [context/specs/auth/_overview.md, context/specs/auth/jwt-service.md]
---
```

## 6. File structure and ownership

### 6.1 Three ownership zones

**Developer/team owned (tool이 절대 건드리지 않음):**
- `CLAUDE.md`
- `CLAUDE.local.md`

**Tool owned (cc-upgrade로 업데이트 가능):**
- `.claude/rules/codified-context.md`
- `.claude/skills/cc-*.md`
- `.claude/agents/` 중 onboarding pipeline agents
- `tools/mcp-server/`
- `.mcp.json`

**Generated → developer owned (초기 생성 후 개발자 소유):**
- `context/**`
- `.claude/agents/` 중 persistent/custom agents

### 6.3 CLAUDE.md와 codified-context.md의 관계

CLAUDE.md = "이 프로젝트가 뭔지". codified-context.md = "context 시스템을 어떻게 쓰는지". 관심사가 다르므로 파일이 다르다.

`cc-onboard` 완료 후, CLAUDE.md에 아래 블록을 **제안** (직접 쓰지 않음):

```markdown
## Context infrastructure
이 프로젝트는 codified-context 시스템을 사용합니다.
- Knowledge base: context/
- Tool integration: .claude/rules/codified-context.md
- MCP server: tools/mcp-server/
- Agent specs: .claude/agents/
```

### 6.4 Subdirectory CLAUDE.md와 _overview.md 역할 분리

`src/auth/CLAUDE.md` (Claude Code 자동 로드) = "이 디렉토리에서 코드 쓸 때의 규칙" (convention).
`context/specs/auth/_overview.md` (MCP로 조회) = "이 모듈이 어떻게 동작하는지" (spec).

용도가 다르므로 공존 가능. Subdirectory CLAUDE.md에서 overview를 참조하도록 연결:

```markdown
# src/auth/CLAUDE.md
이 모듈의 상세 spec: context/specs/auth/_overview.md 참조
관련 decisions: context/decisions/001-chose-local-auth.md 참조
```

## 7. Seamless Claude Code integration

### 7.1 codified-context.md (rules file)

`.claude/rules/codified-context.md`에 `paths` frontmatter 없이 배치하면 매 세션 자동 로드:

```markdown
---
# paths 없음 = 매 세션 자동 로드
---

## Codified context integration

### Pre-change protocol
파일 수정 전:
- mcp__spec_infra__suggest_context(path) 호출
- stale 경고가 있으면 반드시 확인 후 진행
- confidence: draft인 문서는 참고만, 반드시 소스 코드와 대조

### Post-change protocol
파일 수정 후:
- mcp__spec_infra__scan_drift() 호출
- 영향받은 context 문서 업데이트 필요 시 context-gardener 호출

### Context type별 행동 규칙
- decision (accepted) → 이 결정을 존중. 뒤집지 마라.
- decision (deprecated/superseded) → 더 이상 유효하지 않음.
- issue → 이 문제를 악화시키지 마라.
- spec → 이 패턴을 따르라.
- convention → 항상 이 규칙을 적용.
- roadmap (planned/in-progress) → 이 방향을 의식하고 작업.
- glossary → 이 용어를 이 의미로 사용.

### Hallucination 방지
context/ 문서에 명시되지 않은 아키텍처적 주장을 하지 마라.
확신이 없으면 "이 부분은 spec에 없습니다" 라고 말하고 소스를 읽어라.

### Compact 시
/compact 시 현재 작업 중인 context 문서 경로를 요약에 포함할 것.

### Agent routing trigger table
[cc-onboard가 프로젝트 분석 후 자동 생성]
```

### 7.2 Hooks integration (defense in depth)

Instruction following은 100%가 아니다. 빠지면 시스템이 silent하게 degrade한다. `PreToolUse` hook으로 이중 방어:

```json
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "command": "tools/mcp-server/drift-warning.sh \"$TOOL_INPUT\""
      }
    ]
  }
}
```

`drift-warning.sh`는 LLM 호출 없이 동작:
1. 수정 대상 파일 경로 추출
2. 관련 spec frontmatter에서 `last_commit` 확인
3. 해당 파일이 `last_commit` 이후 변경됐으면 경고 주입

### 7.3 Session lifecycle

```
세션 시작
  └─ CLAUDE.md + CLAUDE.local.md + .claude/rules/codified-context.md 자동 로드

파일 수정 시도
  ├─ [Hook] drift-warning.sh → stale spec 경고 주입 (항상 동작)
  └─ [Instruction] suggest_context() → 관련 context 목록 + 경고

코드 변경 완료
  └─ [Instruction] scan_drift() → 영향받은 spec → gardener 호출 제안

/compact 시
  └─ 현재 작업 중인 spec 경로를 요약에 포함

세션 종료
  └─ 다음 세션에서 MCP가 git diff로 자동 계산
```

## 8. Onboarding pipeline

### 8.1 Phase structure (Define → Parallel → Integration)

```
Phase 1: 자동 분석 (에이전트만)
  ┌─ structure-scanner
  │    → specs/_overview.md draft (모듈 경계 + 목록)
  │
  ├─ module-analyst ×N (parallel)
  │    → specs/{module}/_overview.md draft
  │    → 필요시 specs/{module}/{component}.md draft
  │
  └─ verification-agent
       → spec의 각 claim을 소스 코드와 cross-validate
       → 불일치 리포트 생성

Phase 2: 인터뷰 (사람 + 에이전트, blocking)
  ┌─ context-interviewer: project.md
  ├─ context-interviewer: decisions/*.md
  ├─ context-interviewer: issues/*.md
  └─ context-interviewer: roadmap/*.md

Phase 3: 종합 (에이전트만)
  ┌─ relationship-mapper → _overview.md들의 dependency 채움
  ├─ convention-extractor → conventions/*.md 생성
  └─ constitution-assembler → trigger table + CLAUDE.md 블록 제안
```

Phase 2는 blocking이 의도적. "코드 분석 끝났습니다. 몇 가지 여쭤볼게요"가 자연스러운 UX.

### 8.2 Checkpoint & resume

산출물 파일 자체가 checkpoint. `cc-onboard --resume`:

```
1. specs/_overview.md 존재? → structure-scanner skip
2. specs/{module}/_overview.md 각각 존재? → 있는 것 skip
3. verification report 존재? → skip
4. project.md 존재? → 인터뷰 skip (--force-interview로 재실행 가능)
5. conventions/ 존재? → skip
6. trigger table 존재? → skip
```

### 8.3 Incremental onboarding

```
cc-onboard --incremental
  1. structure-scanner 재실행 → 새 모듈 감지
  2. 새 모듈에 대해서만 module-analyst 실행
  3. relationship-mapper가 기존 그래프에 추가
  4. trigger table 업데이트
```

## 9. Hallucination mitigation

### 9.1 문제 분류

**Onboarding hallucination:** 코드 분석 시 존재하지 않는 패턴을 "발견"하거나 잘못된 추론. 50K LOC를 context window에 한 번에 넣을 수 없으므로 본질적으로 발생 가능.

**Runtime hallucination:** 에이전트가 spec에 없는 내용을 "있는 것처럼" 주장.

### 9.2 대응 전략

**A. Confidence 필드:** Onboarding 생성 문서는 전부 `confidence: draft`. 개발자 리뷰 후 `reviewed`, 테스트 검증 후 `verified`. Instruction: "draft 문서는 참고만, 반드시 소스 대조."

**B. Evidence 강제:** spec-writer/module-analyst가 모든 기술적 claim에 소스 경로+라인 citation:

```markdown
## Token rotation
RefreshTokenService (src/auth/refresh.ts:42-67) handles rotation.
Expiry is 1 hour (src/auth/config.ts:8).
```

**C. Cross-validation (verification-agent):**

```
✓ "AuthService uses JWT"           → confirmed (src/auth/service.ts:15)
✗ "Token expiry is 24h"            → ACTUAL: 1h (src/auth/config.ts:8)
? "Supports refresh rotation"      → UNCERTAIN
```

NOT FOUND → spec에서 제거/수정. UNCERTAIN → confidence 하향 + 인터뷰 Phase에서 확인.

**D. Runtime instruction:** codified-context.md에 포함. Section 7.1 참조.

## 10. Context interviewer

### 10.1 Agent 설계 원칙

```
- 한 번에 질문 1-2개만. 폭격하지 마라.
- 답변을 즉시 문서 draft로 변환해서 보여줘라.
- 개발자가 "맞아" 하면 저장, "아니" 하면 수정.
- 추측하지 마라. 모르면 물어봐라.
- 코드 분석 결과와 모순되면 명시적으로 확인:
  "코드 분석에서는 X로 보이는데, Y가 맞나요?"
```

### 10.2 인터뷰 flow (type별)

**project.md:** 핵심 문제 → 사용자/고객 → 기술적 제약.
**decisions:** 결정 내용 → 검토한 대안 → trade-off.
**issues:** 기술 부채 → 영향 범위 → 우회 방법.
**roadmap:** 주요 계획 → 영향 모듈 → 마이그레이션 순서.

### 10.3 Ad-hoc 인터뷰 (skills 연동)

```
cc-add-decision  → context-interviewer 호출
cc-add-roadmap   → context-interviewer 호출
cc-add-spec      → spec-writer + context-interviewer
cc-add-issue     → context-interviewer + code-reviewer
```

예: `cc-add-decision` 후 "PostgreSQL에서 MongoDB로 바꾸기로 했어" → 인터뷰어가 대안/trade-off 질문 → ADR 생성 → 이전 decision에 superseded 반영.

## 11. Drift detection

### 11.1 Spec drift (자동, MCP 기반)

`scan_drift()`: LLM 없이 순수 git operation.

```
1. context/ 내 모든 .md frontmatter 파싱
2. covers의 각 파일에 대해 git log --since={last_synced}
3. last_commit 이후 변경 있으면 → stale 판정
4. depends_on의 다른 문서가 stale이면 → cascade stale
5. 결과: stale 문서 목록 + cascade chain + 영향 범위
```

### 11.2 Decision/convention drift (주기적, agent 기반)

Decisions/conventions에는 `covers` 없으므로 git diff만으론 불가. context-gardener의 주기적 검증:
- `cc-drift-check --deep`으로 decisions/conventions를 코드와 대조
- 불일치 발견 시 리포트 + 업데이트 제안

### 11.3 Hook 기반 실시간 경고

Section 7.2 참조. `PreToolUse` hook으로 파일 수정 전 자동 경고 주입.

## 12. Git worktree 전략

### 12.1 Worktree에서의 파일 가용성

Git worktree는 tracked file의 독립 복사본을 가진다. 우리 시스템의 파일 분류:

| 파일 | Tracked? | Worktree에서 자동 가용? |
|------|----------|----------------------|
| `CLAUDE.md` | Yes | ✅ 자동 |
| `CLAUDE.local.md` | No (gitignored) | ❌ .worktreeinclude 필요 |
| `.claude/settings.json` | Yes | ✅ 자동 |
| `.claude/agents/`, `rules/`, `skills/` | Yes | ✅ 자동 (단, 알려진 버그 존재) |
| `.mcp.json` | Yes | ✅ 자동 |
| `context/**` | Yes | ✅ 자동 |
| `tools/mcp-server/` | Yes | ✅ 자동 |

**핵심:** `context/`, `.claude/`, `.mcp.json`, `tools/`가 전부 tracked이므로 worktree에서 자동으로 가용하다. MCP 서버의 상대 경로(`tools/mcp-server/server.py`)도 worktree root에서 정상 resolve.

### 12.2 .worktreeinclude 설정

`cc-init`이 생성해야 할 `.worktreeinclude`:

```
CLAUDE.local.md
.env
.env.*
**/.claude/settings.local.json
```

### 12.3 Context 문서 conflict 전략

Worktree 간 같은 context 문서를 동시 수정하면 merge 시 conflict 발생 가능. 특히 위험한 케이스:

**Frontmatter 자동 필드 conflict:** `last_commit`, `last_synced`를 두 worktree에서 각각 업데이트하면 conflict. → 아무 쪽이든 accept 후 `cc-drift-check` 실행. 자동 재계산됨.

**Spec 내용의 의미적 conflict:** Git이 auto-merge를 성공시키지만 논리적으로 모순인 경우 (e.g., 한쪽이 "만료 1시간", 다른 쪽이 "리프레시 불필요"). → Code conflict와 동일하게 수동 resolve 후 `context-gardener`로 코드 ↔ spec 정합성 재검증.

**Conflict 예방 원칙:**
- Spec은 가능한 작은 단위(component level)로 쪼갠다. → 각 branch가 다른 파일을 건드릴 확률 상승.
- 한 branch에서 spec을 수정하면 해당 spec의 `covers` 영역만 수정한다.
- ADR/roadmap은 append-only(새 파일 추가)를 기본으로 한다. → 같은 파일 동시 수정 방지.
- Conventions는 섹션 단위 분리를 권장 (`error-handling.md`, `testing.md` 등).

### 12.4 MCP 서버 격리

각 worktree session은 독립적인 MCP 서버 인스턴스를 시작. `context/`가 worktree마다 독립 복사본이므로, 각 MCP 서버는 해당 worktree의 context 상태를 반영. Branch 간 spec 차이가 있어도 각 세션이 자기 branch의 context만 참조.

## 13. Versioning and upgrade


### 13.2 cc-upgrade workflow

```
cc-upgrade
  1. 현재 tool version 확인
  2. GitHub repo에서 최신 버전 fetch
  3. Tool layer만 diff 표시
  4. Project layer 절대 보존
  5. codified-context.md에 새 instruction → 머지 제안
  6. 새 agent 템플릿 → "추가할까요?" 확인
  7. MCP server 업데이트
  8. Breaking change → migration guide 출력
```

네트워크 불가 시: `cc-upgrade --from ./path/to/release/`

## Appendix A: Design decisions log

이 문서 작성 과정에서 내려진 주요 설계 결정 기록.

| # | 결정 | 이유 |
|---|------|------|
| D1 | MCP는 cold memory serving이 아닌 lightweight utility | spec 파일은 그냥 .md로 프로젝트에 저장. MCP가 serving할 필요 없음. |
| D2 | 전체 문서를 MCP 메모리에 보유 | 100개 문서 ≈ 400KB. In-memory 전량 보유가 가장 단순하고 빠름. |
| D3 | .claude/rules/에 시스템 instruction 배치 | CLAUDE.md 소유권 충돌 방지. 개발자의 CLAUDE.md를 건드리지 않음. |
| D4 | Roadmap에 invalidates 대신 related_specs | 미래 예측의 strong claim은 위험. Soft warning으로 전환. |
| D5 | depends_on은 module overview에서만 | 유지보수 범위 폭발 방지. Tags 기반 검색으로 보완. |
| D6 | Frontmatter 자동 필드 (last_commit, last_synced) | 사람의 유지보수 부담 최소화. Tool이 자동 관리. |
| D7 | Onboarding Phase 2를 blocking으로 설계 | project.md, ADR, roadmap은 코드에 없는 정보. 인터뷰 필수. |
| D8 | suggest_context/search_context 분리 유지 | "파일 기반 조회"와 "개념 기반 탐색"은 의도가 다름. |
| D9 | Spec을 계층적 디렉토리로 구성 | Worktree conflict 분산 + MCP zoom level 지원 + 대규모 프로젝트 대응. |
| D10 | Hook으로 instruction 이중 방어 | Instruction following ≠ 100%. Hook이 물리적으로 경고 삽입. |
