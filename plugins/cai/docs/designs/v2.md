## 1. Problem statement

AI 코딩 에이전트는 세션 간 기억이 없다. 프로젝트의 컨벤션, 아키텍처 결정, 알려진 문제, 미래 계획을 매번 다시 설명해야 하며, 설명이 빠지면 에이전트가 과거 실수를 반복하거나 프로젝트 방향에 반하는 코드를 생성한다.

단일 CLAUDE.md 파일은 소규모 프로젝트에서는 동작하지만, 50K+ LOC 규모에서는 한계에 도달한다. Vasilopoulos(arXiv:2602.20478)는 108K LOC C# 프로젝트에서 3-tier codified context 아키텍처를 실증했으나, 단일 개발자/단일 프로젝트/단일 도메인에 특화되어 있다.

본 설계는 이 아키텍처를 범용화한다. 10K~1M+ LOC, 1~10+인 팀, Claude Code + Codex, 언어/스택 무관하게 적용 가능한 codified context infrastructure를 정의한다.

## 2. Design goals

- **Seamless**: 개발자는 이 프레임워크의 존재를 인지만 하는 수준. 나머지는 AI-driven.
- **Incremental**: 처음부터 완벽할 필요 없이, 실패에서 점진적으로 성장.
- **Tool-agnostic outputs**: 산출물(context 문서)은 plain markdown. 어떤 도구든 읽을 수 있음.
- **Ownership clarity**: tool이 관리하는 영역과 개발자가 소유하는 영역이 명확히 분리.
- **AI-driven maintenance**: AI가 자기 기억을 스스로 관리하고, 개발자는 승인만.

## 3. Design principles

### 3.1 v1에서 v2로의 핵심 전환

| v1 | v2 | 이유 |
|----|----|----|
| Skill 중심 워크플로 | Rules file 중심 | 개발자가 명령어를 외우는 건 seamless가 아님 |
| 개발자가 skill 호출 | AI가 자동 행동, 개발자는 승인 | AI를 위한 시스템이니 AI가 주도 |
| MCP 서버 필수 | MCP 없이 시작 | adoption friction 최소화 |
| covers 필드 필수 | optional (convention-based 매핑 우선) | covers 자체가 drift 원인이 될 수 있음 |
| Constitution 별도 파일 | CLAUDE.md + rules로 갈음 | 기존 메커니즘 활용 |

### 3.2 개발자의 실제 경험 (목표)

```
Day 1:
  개발자: "이 프로젝트에 codified context 세팅해줘"
  AI:     (알아서 분석, 몇 가지 질문, 세팅 완료)
  개발자: 질문에 답변만.

Day 2~N:
  개발자: "결제 모듈에 환불 기능 추가해줘"
  AI:     (자동으로 관련 spec 읽고, convention 따르고, 구현)
  AI:     "환불 flow를 spec에 반영할까요?"
  개발자: "ㅇㅇ"

가끔:
  AI:     "auth spec이 좀 오래됐어요. 업데이트할까요?"
  개발자: "ㅇㅇ"

개발자가 외워야 할 명령어: 없음
개발자가 관리해야 할 것: CLAUDE.md (이미 하던 것)
```

## 4. Architecture overview — 3 layers

시스템의 동력은 skill이 아니라 **rules file(instruction)**이다. AI가 매 세션 이 instruction을 읽고 자동으로 행동한다.

### Layer 1: Rules file (엔진)

`.claude/rules/codified-context.md` — 매 세션 자동 로드. AI의 모든 자동 행동을 정의.

### Layer 2: Hook (안전망)

`PreToolUse` hook — instruction following이 100%가 아닌 현실에 대한 물리적 보험. Edit/Write 시 관련 spec이 stale이면 경고 주입. LLM 호출 없음, ms 단위.

### Layer 3: Skills (escape hatch)

개발자가 명시적으로 호출할 수 있는 명령. 대부분 AI가 내부적으로 호출하며, 개발자가 직접 쓸 일은 거의 없음.

| Skill | 누가 호출? | 빈도 |
|-------|----------|------|
| `cc-init` | 개발자 (최초 1회) | 1번 |
| `cc-onboard` | AI가 cc-init 안에서 | 1번 |
| `cc-add-spec` | AI가 자동으로 (post-work 제안 승인 시) | 자주 |
| `cc-add-decision` | AI가 자동으로 (knowledge capture 승인 시) | 가끔 |
| `cc-drift-check` | AI가 자동으로 (stale 감지 시) | 자동 |
| `cc-capture-lesson` | 개발자 (원하면) | 드묾 |
| `cc-upgrade` | 개발자 (버전 업 시) | 드묾 |
| `cc-add-agent` | 개발자 또는 AI | 드묾 |
| `cc-add-roadmap` | AI가 자동으로 (knowledge capture 승인 시) | 드묾 |

### Agents (autonomous workers)

코드를 읽고 판단해서 산출물을 만드는 자율적 행위자. `.claude/agents/`에 markdown으로 정의.

**Onboarding pipeline agents (tool 소유):**

| Agent | 역할 | Phase |
|-------|------|-------|
| `structure-scanner` | 디렉토리/파일 분석, 모듈 경계 식별 | 1 (자동) |
| `module-analyst` | 단일 모듈 내부 구조 분석 (×N 병렬) | 1 (자동) |
| `draft-generator` | git log/blame/코드 분석 → decisions, issues, project.md 초안 자동 생성 | 2 (자동) |
| `context-interviewer` | 초안 검증 + 코드에서 추출 불가능한 지식 보완 인터뷰 | 3 (대화) |
| `relationship-mapper` | 모듈 간 의존 관계, 데이터 흐름 매핑 | 4 (자동) |
| `convention-extractor` | 반복 패턴 추출 (naming, error handling 등) | 4 (자동) |
| `constitution-assembler` | 산출물 종합 → CLAUDE.md 업데이트 제안 | 4 (자동) |

**Persistent agents (생성 후 개발자 소유):**

| Agent | 역할 | 호출 시점 |
|-------|------|----------|
| `verification-agent` | spec의 claim을 소스 코드와 cross-validate. 독립 context window에서 검증. | spec 생성/수정 시마다 |
| `context-gardener` | stale spec 업데이트, 새 패턴 발견 시 spec 제안, dead spec/resolved issue/completed roadmap pruning | post-change |
| `code-reviewer` | convention 위반 탐지, post-change 리뷰 | post-change |
| `spec-writer` | 소스 읽고 spec 작성, 기존 spec과 일관성 검증 | ad-hoc |
| `context-interviewer` | ADR/roadmap 등 사람 지식이 필요한 문서 작성 | ad-hoc |
| `{custom-domain-agents}` | 프로젝트별 domain expert | ad-hoc |

## 5. Constitution: CLAUDE.md + rules file

논문의 Tier 1(Constitution)은 ~660줄의 항상 로드되는 핵심 규칙. v2에서는 별도 파일 없이 기존 메커니즘으로 구현한다.

### 역할 분리

```
CLAUDE.md (개발자 소유)             codified-context.md (tool 소유)
─────────────────────               ─────────────────────────────
프로젝트가 뭔지 (what)              context 시스템을 어떻게 쓸지 (how)
프로젝트 개요, tech stack            pre/post-change protocol
빌드/테스트 명령어                   context type별 행동 규칙
핵심 conventions 요약               trigger table (선택적)
아키텍처 요약 + context/ 링크        knowledge capture protocol
critical issues 요약                hallucination 방지 instruction
```

둘을 합쳐서 논문의 Constitution에 대응한다. CLAUDE.md는 **요약 + 링크**, 상세 내용은 context/에.

### Tool의 CLAUDE.md 관여 방식

Tool은 CLAUDE.md를 **직접 수정하지 않는다**. `cc-onboard` 완료 시 또는 constitution-assembler가 conventions/issues 변경 감지 시, "CLAUDE.md에 이 블록을 추가/업데이트하시겠습니까?"로 **제안**만 한다.

### CLAUDE.md 예시 (cc-onboard 후)

```markdown
# CLAUDE.md

## Project
결제 플랫폼. Node.js + PostgreSQL + Redis.

## Build & Test
pnpm build / pnpm test

## Key Conventions (요약)
- 에러 처리: domain-specific error class 사용 (context/conventions/error-handling.md 참조)
- API 설계: REST, snake_case (context/conventions/api-design.md 참조)

## Architecture (요약)
- 3개 모듈: auth, payment, notification (context/specs/_overview.md 참조)

## 도메인 용어
핵심 도메인 용어: context/glossary.md 참조

## Critical Issues
- N+1 쿼리 문제 미해결 (context/issues/perf-n-plus-one.md 참조)

## Context Infrastructure
이 프로젝트는 codified context를 사용합니다.
- Knowledge base: context/
- Rules: .claude/rules/codified-context.md
```

## 6. Rules file: AI 자동 행동 정의

`.claude/rules/codified-context.md`는 매 세션 자동 로드되어 AI의 행동을 규정하는 핵심 엔진.

```markdown
# .claude/rules/codified-context.md (초안)

## Pre-work: Context loading
파일을 수정하기 전, 해당 파일과 관련된 context 문서가 있는지 확인하라.
- context/specs/ 에서 디렉토리 구조 기반으로 관련 spec 탐색
- 발견하면 읽고 패턴을 따르라
- confidence: draft/intent인 문서는 참고만, 소스 코드와 대조

## Post-work: Spec maintenance
코드 변경 완료 후:
- 변경이 기존 spec의 내용과 달라졌으면 → "이 변경을 spec에 반영할까요?" 제안
- 관련 spec이 없는 영역에 비자명한 패턴을 만들었으면 → "새 spec으로 문서화할까요?" 제안
- 제안은 작업 완료 후 한 번만. 작업 중에 끼어들지 마라.

## Knowledge capture protocol
작업 중 다음을 발견하면, 작업 흐름을 끊지 말고 완료 시점에 제안:
1. 새 failure mode (디버깅으로 root cause를 찾았을 때)
   → "이 failure mode를 [agent/issue]에 기록할까요?"
2. 비자명한 패턴 ("이렇게 해야 동작한다"를 발견했을 때)
   → "이 패턴을 [convention/spec]에 추가할까요?"
3. 아키텍처 결정 (대안을 비교하고 하나를 선택했을 때)
   → "이 결정을 decision으로 기록할까요?"

제안 형식: 한 줄 요약 + 승인 요청. 장황하게 설명하지 마라.
개발자가 "아니"라고 하면 즉시 넘어가라.

## Stale awareness
Hook이 stale 경고를 주입하면:
- 해당 spec을 읽되, 현재 코드와 다른 부분은 spec보다 코드를 신뢰
- 작업 완료 후 "이 spec이 outdated입니다. 업데이트할까요?" 제안

## Context type별 행동 규칙
- decision (accepted) → 이 결정을 존중. 뒤집지 마라.
- decision (deprecated/superseded) → 더 이상 유효하지 않음. superseded_by 참조.
- issue → 이 문제를 악화시키지 마라.
- spec → 이 패턴을 따르라.
- convention → 항상 이 규칙을 적용.
- roadmap (planned/in-progress) → 이 방향을 의식하고 작업.
- glossary → 이 용어를 이 의미로 사용.

## Hallucination 방지
context/ 문서에 명시되지 않은 아키텍처적 주장을 하지 마라.
확신이 없으면 "이 부분은 spec에 없습니다" 라고 말하고 소스를 읽어라.

## /compact 시
현재 작업 중인 context 문서 경로를 요약에 포함할 것.

## Action routing
제안이 승인되면 아래 테이블의 skill을 호출하라.

| 상황 | Skill |
|------|-------|
| spec 생성/수정 필요 | cc-add-spec |
| 새 decision 발견 | cc-add-decision |
| failure mode/패턴 발견 | cc-capture-lesson |
| 새 roadmap 항목 발견 | cc-add-roadmap |
| stale spec 업데이트 필요 | cc-drift-check |
| 새 agent 필요 | cc-add-agent |

## [PROJECT-SPECIFIC] Agent routing trigger table
[cc-add-agent 시 점진적으로 축적. 없어도 동작함.]
```

## 7. Knowledge taxonomy

프로젝트 지식을 시제(temporal axis)로 분류한다. 각 type은 에이전트에게 서로 다른 행동을 요구한다.

### 7.1 Types and agent behavior

| 시제 | Type | 에이전트 행동 |
|------|------|--------------|
| 정체성 | `project` | 프로젝트의 목적과 제약을 이해하고 작업 |
| 과거 | `decision` (accepted) | 이 결정을 존중하라. 뒤집지 마라. |
| 과거 | `decision` (deprecated/superseded) | 더 이상 유효하지 않음. superseded_by 참조. |
| 현재 | `spec` | 이 패턴을 따르라. |
| 현재 | `convention` | 항상 이 규칙을 적용하라. |
| 현재 | `issue` | 이 문제를 악화시키지 마라. |
| 현재 | `glossary` | 이 용어를 이 의미로 사용하라. CLAUDE.md와 관련 spec에서 링크로 참조. 별도 로드 메커니즘 없이, 이미 로드되는 문서의 링크를 통해 자연스럽게 접근. |
| 미래 | `roadmap` (planned/in-progress) | 이 방향을 의식하고 작업하라. 대규모 투자 자제. |
| 미래 | `roadmap` (exploring) | 참고만. 확정되지 않음. |

### 7.2 Directory structure

```
context/
├── project.md                          ← 프로젝트 정체성, 배경, 비전
├── glossary.md                         ← 도메인 용어 정의 (Ubiquitous Language)
│
├── decisions/                          ← ADR (시간순, flat, append-only)
│   ├── 001-chose-postgresql.md
│   ├── 002-event-driven-arch.md
│   └── 003-monorepo-structure.md
│
├── issues/                             ← Known issues / tech debt (flat)
│   ├── perf-n-plus-one.md
│   └── legacy-auth-migration.md
│
├── conventions/                        ← 코딩 컨벤션 / 패턴 (flat)
│   ├── error-handling.md
│   ├── testing-strategy.md
│   └── api-design.md
│
├── specs/                              ← 서브시스템 기술 스펙 (계층적)
│   ├── _overview.md                    ← 전체 아키텍처, 모듈 간 관계
│   ├── auth/
│   │   ├── _overview.md                ← auth 모듈 설계
│   │   ├── jwt-service.md              ← 컴포넌트 레벨
│   │   └── oauth-provider.md
│   ├── payment/
│   │   ├── _overview.md
│   │   ├── stripe-integration.md
│   │   └── refund-flow.md
│   └── notification/
│       ├── _overview.md
│       └── email-sender.md
│
└── roadmap/                            ← 미래 계획 (status별)
    ├── planned/
    │   └── migrate-to-oauth2.md
    ├── exploring/
    │   └── graphql-migration.md
    └── completed/
        └── initial-launch.md
```

`_overview.md`가 각 계층의 접착제 역할. Project-level overview는 모듈 간 관계를, module-level overview는 내부 컴포넌트 간 관계를 담는다.

### 7.3 context/ 디렉토리명 충돌 대응

일부 프로젝트는 `context/`를 다른 용도로 사용할 수 있다. `codified-context.md` 설정에서 루트 디렉토리명과 소스 루트를 변경 가능:

```markdown
<!-- .claude/rules/codified-context.md 상단 -->
Context directory: .context
Source roots: [src/, packages/*/src/]
```

**Context directory** 기본값은 `context/`. 대안: `.context`, `docs/context`, `knowledge` 등.

**Source roots**는 convention-based 매핑의 기준점. 기본값은 `[src/]`. 모노레포나 소스 루트가 `src/`가 아닌 프로젝트에서 설정. cc-onboard 시 자동 감지하여 초기값 설정.

```
# Source roots: [src/, packages/*/src/] 일 때 convention-based 매핑:
context/specs/auth/_overview.md → src/auth/** 또는 packages/*/src/auth/**
context/specs/auth/jwt-service.md → src/auth/jwt*.ts 또는 packages/*/src/auth/jwt*.ts
```

## 8. Frontmatter schema

### 8.1 설계 원칙: 경량화

Frontmatter 필드가 많아지면 유지보수 부담이 커지고, 이는 논문의 핵심 실패 모드(stale spec)와 동일한 문제를 야기한다. 공통 필수 필드를 최소화하고, type/level별 선택 필드를 분리한다. 자동 관리 가능한 필드는 tool이 관리.

### 8.2 Field definitions

**공통 필수 (모든 context 문서):**

| Field | Type | 설명 | 관리 주체 |
|-------|------|------|----------|
| `type` | enum | decision, issue, spec, convention, roadmap, glossary, project | 생성 시 설정 |
| `tags` | string[] | 키워드 태그 (검색 1순위 인덱스) | 사람 |
| `last_synced` | date | spec과 코드가 마지막으로 동기화된 날짜. spec 생성/수정 시 현재 시각으로 갱신. | tool (자동) |

**spec 전용:**

| Field | Type | 설명 | 관리 주체 |
|-------|------|------|----------|
| `level` | enum | project, module, component | 생성 시 설정 |
| `covers` | string[] | 이 spec이 다루는 소스 파일 glob 패턴 | **optional** (없으면 convention-based 매핑) |
| `parent` | string | 상위 overview 경로 | tool (자동) |
| `confidence` | enum | intent, draft, reviewed, verified | 사람 |

**module overview 전용 (추가):**

| Field | Type | 설명 |
|-------|------|------|
| `components` | string[] | 하위 컴포넌트 목록 |
| `exports` | string[] | 다른 모듈이 의존하는 public interface |
| `depends_on` | string[] | 의존하는 다른 모듈/문서 |

**project overview 전용 (추가):**

| Field | Type | 설명 |
|-------|------|------|
| `modules` | string[] | 전체 모듈 목록 |
| `module_dependencies` | map | 모듈 간 의존 관계 |

**decision 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `status` | enum | proposed, accepted, deprecated, superseded |
| `superseded_by` | string | 대체 decision 경로 |

**roadmap 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `status` | enum | exploring, planned, in-progress, completed |
| `target` | string | 목표 시기 (e.g., 2026-Q3) |
| `related_specs` | string[] | 영향받을 수 있는 spec (soft link) |

**issue 전용:**

| Field | Type | 설명 |
|-------|------|------|
| `severity` | enum | low, medium, high, critical |

### 8.3 Confidence 필드 (v2 추가: intent)

| 값 | 의미 | 생성 시점 |
|---|------|---------|
| `intent` | 아직 구현 안 됨, 이렇게 할 예정 | 새 프로젝트 onboarding |
| `draft` | 코드 분석으로 자동 생성, 미검증 | 기존 프로젝트 onboarding |
| `reviewed` | 개발자가 읽고 확인 | 개발자 리뷰 후 |
| `verified` | 테스트/실행으로 검증 | 테스트 통과 후 |

Agent 행동: `intent`는 "목표 설계"로 참고하되 실제 코드와 다를 수 있음을 인지. `draft`는 참고만, 반드시 소스 대조.

### 8.4 covers 필드: optional + convention-based 매핑

`covers`가 없을 때 convention-based 매핑으로 관련 소스를 추정:

```
context/specs/auth/_overview.md   →  src/auth/** (디렉토리 구조 기반)
context/specs/auth/jwt-service.md →  src/auth/jwt*.ts (이름 기반 heuristic)
```

Convention 매핑이 실패할 때만 `covers`를 명시적으로 작성한다. 이렇게 하면 리팩토링으로 파일이 이동해도 covers가 깨지는 문제를 완화할 수 있다.

### 8.5 예시

```yaml
# specs/auth/_overview.md
---
type: spec
level: module
tags: [auth, security, jwt]
last_synced: 2026-04-02
confidence: reviewed
components: [jwt-service, oauth-provider]
exports: [AuthService.validateToken(), AuthService.createSession()]
depends_on: [context/specs/payment/_overview.md]
---
```

```yaml
# roadmap/planned/migrate-to-oauth2.md
---
type: roadmap
status: planned
tags: [auth, security, migration]
target: 2026-Q3
related_specs: [context/specs/auth/_overview.md, context/specs/auth/jwt-service.md]
---
```

## 9. Agent domain knowledge

### 9.1 논문의 핵심 발견

Agent spec의 >50%는 행동 지침이 아니라 도메인 지식이다. network-protocol-designer의 915줄 중 ~600줄이 determinism theory, hash function 제약, failure mode 등. 이 pre-loaded knowledge가 agent의 전문성을 만든다.

### 9.2 Agent spec 표준 구조

```markdown
# {agent-name}

## Role & Scope
(역할, 도구, 권한 — 행동 지침)

## Domain Knowledge
(이 에이전트가 알아야 할 프로젝트 도메인 지식. 시간이 지나며 축적됨.)

### Architecture
- 이 도메인의 핵심 구조

### Code Patterns
- 따라야 할 패턴 (코드 예시 포함)

### Failure Modes
| Symptom | Cause | Fix |
|---------|-------|-----|
(디버깅할 때마다 행이 추가됨)

### Decision Context
- 관련 ADR 요약 + 링크

## Referenced Specs
(필요시 읽어야 할 cold memory 문서 목록)
```

### 9.3 축적 메커니즘: Emergence pattern

논문의 실제 방식을 시스템적으로 지원:

```
1. 범용 에이전트로 작업 시도
2. 같은 도메인에서 반복 실패 발생
3. 실패 원인과 해결책을 에이전트 spec에 축적
4. 다음 세션부터 해당 에이전트가 pre-loaded knowledge로 활용
```

축적은 두 가지 경로로 발생:

**A. 실시간 (rules file instruction 기반):**
- AI가 작업 중 failure mode/pattern/decision을 발견
- 작업 완료 시점에 "이걸 agent spec에 기록할까요?" 제안
- 승인 시 Failure Modes 테이블에 행 추가

**B. 회고 (cc-capture-lesson skill):**
- 개발자가 세션 중 아무 때나 `/cc-capture-lesson` 호출
- AI가 세션을 돌아보며 문서화할 지식 제안
- 승인 시 적절한 context 문서에 반영

### 9.4 Symptom-Cause-Fix 테이블

논문에서 가장 실용적이었던 포맷. `issue` type과 agent spec의 Failure Modes 섹션에서 표준으로 사용:

```markdown
## Known Failure Modes
| Symptom | Cause | Fix |
|---------|-------|-----|
| Webhook 재전송 반복 | refund 동기처리 >30s | 비동기 큐 전환 |
| Desync on crits | Using local time | GetSyncedTime() |
```

## 10. Drift detection

### 10.1 Stale spec: primary failure mode

논문에서 stale spec이 primary failure mode. 에이전트는 문서를 절대적으로 신뢰하므로, outdated spec은 없는 spec보다 위험하다. v2는 3단계 방어를 구축한다.

### 10.2 방어 1: Hook (물리적, 실시간)

Node.js/Python 스크립트로 구현. 대부분의 프로젝트에 이미 런타임이 있고, YAML frontmatter 파싱이 견고함.

```javascript
// tools/drift-warning.js — PreToolUse hook (Edit/Write 시 실행)
// 수정 대상 파일 경로를 받아, 관련 spec이 stale인지만 체크

// 1. context/specs/ 에서 covers에 이 파일이 포함된 문서 찾기
//    (또는 convention-based 매핑, Source roots 기반)
// 2. 해당 문서의 last_synced 이후 FILE_PATH가 변경됐는지 (git log --since)
// 3. 변경됐으면 stderr로 경고 출력
```

Hook 등록:

```json
// .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit",
        "command": "node tools/drift-warning.js \"$TOOL_INPUT\""
      }
    ]
  }
}
```

### 10.3 방어 2: Rules instruction (AI 자발적)

코드 변경 완료 후 AI가 스스로 spec 업데이트를 제안. Section 6 참조.

### 10.4 방어 3: cc-drift-check (전체 스캔)

```
/cc-drift-check 실행 흐름:
1. context/ 내 모든 .md 파일의 frontmatter 파싱
2. 각 문서에 대해 stale 여부 판정 (3단계)
3. 결과 리포트 출력
4. 업데이트 필요한 문서에 대해 action 제안
```

**Stale 판정 로직:**

```
문서에 covers 필드가 있는가?
├── YES → covers의 파일들에 대해 git log --since="{last_synced}"
│         변경 있으면 → STALE (확실)
│
└── NO → convention-based 매핑 시도 (Source roots 기반)
    ├── context/specs/auth/_overview.md → {source_roots}/auth/** 추정
    │   변경 있으면 → POSSIBLY STALE (추정)
    │
    └── 매핑 불가 (decisions, conventions 등)
        → last_synced가 N일 이상 → REVIEW SUGGESTED
```

**리포트 포맷:**

```
📋 Drift Check Report (2026-04-06)

🔴 STALE (code changed, spec outdated):
  context/specs/payment/stripe-integration.md
    covers: src/payment/stripe/**
    changed files: src/payment/stripe/webhook.ts (+45 -12)
    last_synced: 2026-03-28 (9 days ago)

🟡 POSSIBLY STALE (convention-based match):
  context/specs/auth/_overview.md
    inferred: src/auth/**
    changed files: src/auth/refresh.ts (+8 -3)
    last_synced: 2026-04-01 (5 days ago)

🟢 REVIEW SUGGESTED (time-based):
  context/decisions/002-event-driven-arch.md
    last_synced: 2026-02-15 (50 days ago)
```

### 10.5 "두 번 설명했으면 문서화하라" (G4) — Drift의 부산물로 구현

cc-drift-check가 "spec 없이 자주 수정되는 파일 영역"을 감지:

```
⚠ src/payment/webhook.ts — 최근 5 세션에서 수정, 관련 spec 없음
→ spec 생성을 권장합니다
```

사람의 기억에 의존하지 않고, git history라는 객관적 데이터에 기반.

## 11. Onboarding

### 11.1 기존 프로젝트 (코드 → 분석 → 초안 → 검증 인터뷰)

```
Phase 1: 자동 분석 — spec 생성 (에이전트만)
  ┌─ structure-scanner
  │    → specs/_overview.md draft (모듈 경계 + 목록)
  │
  ├─ module-analyst ×N (parallel)
  │    → specs/{module}/_overview.md draft
  │    → 필요시 specs/{module}/{component}.md draft
  │
  └─ verification-agent
       → spec의 각 claim을 소스 코드와 cross-validate
       → 불일치 리포트 생성

Phase 2: 자동 초안 생성 — 코드 외 문서 (에이전트만)
  ┌─ draft-generator
  │    ├─ git log/blame 분석 → decisions/*.md 초안
  │    ├─ 코드 분석 (TODO, FIXME, 복잡도, 반복 패턴) → issues/*.md 초안
  │    ├─ README, docs/ 분석 → project.md 초안
  │    └─ 모든 초안은 confidence: draft

Phase 3: 검증 인터뷰 (사람 + 에이전트, blocking)
  └─ context-interviewer
       ├─ 초안을 보여주며 검증: "git history 보니 이런 결정들이 보이는데, 맞나요?"
       ├─ 코드에서 추출 불가능한 지식 보완: 비전, 제약, 미래 계획
       └─ roadmap/*.md 작성 (코드에서 추론 불가)

Phase 4: 종합 (에이전트만)
  ┌─ relationship-mapper → _overview.md들의 dependency 채움
  ├─ convention-extractor → conventions/*.md 생성
  └─ constitution-assembler → CLAUDE.md 업데이트 제안
```

### 11.2 새 프로젝트 (인터뷰 → 문서 → 코드 → 검증)

```
cc-init --new
  1. 디렉토리 구조 생성
  2. context-interviewer 즉시 시작:
     - "무엇을 만드나요?" → project.md
     - "기술 스택은?" → project.md + conventions/ 초안
     - "핵심 아키텍처 결정은?" → decisions/*.md
     - "알고 있는 위험/제약은?" → issues/*.md
  3. 초기 constitution 조립 (CLAUDE.md 업데이트 제안)
  4. 모듈 구조가 생기면 cc-onboard --incremental로 spec 생성
```

새 프로젝트에서 생성된 spec은 `confidence: intent` (아직 구현 안 됨). 구현 완료 시 `draft`로 전환.

### 11.3 Checkpoint & resume

산출물 파일 자체가 checkpoint. `cc-onboard --resume`:

```
1. specs/_overview.md 존재? → structure-scanner skip
2. specs/{module}/_overview.md 각각 존재? → 있는 것 skip
3. verification report 존재? → skip
4. project.md 존재? → 인터뷰 skip (--force-interview로 재실행 가능)
5. conventions/ 존재? → skip
```

### 11.4 Incremental onboarding

```
cc-onboard --incremental
  1. structure-scanner 재실행 → 새 모듈 감지
  2. 새 모듈에 대해서만 module-analyst 실행
  3. relationship-mapper가 기존 그래프에 추가
  4. CLAUDE.md 업데이트 제안
```

### 11.5 Hallucination 대응

**A. Confidence 필드:** 자동 생성 문서는 전부 `confidence: draft`. Agent instruction: "draft 문서는 참고만, 반드시 소스 대조."

**B. Evidence 강제:** spec-writer/module-analyst가 모든 기술적 claim에 소스 경로+라인 citation:

```markdown
## Token rotation
RefreshTokenService (src/auth/refresh.ts:42-67) handles rotation.
Expiry is 1 hour (src/auth/config.ts:8).
```

**C. Cross-validation (verification-agent):**

```
✓ "AuthService uses JWT"       → confirmed (src/auth/service.ts:15)
✗ "Token expiry is 24h"        → ACTUAL: 1h (src/auth/config.ts:8)
? "Supports refresh rotation"  → UNCERTAIN
```

NOT FOUND → spec에서 제거/수정. UNCERTAIN → confidence 하향 + 인터뷰에서 확인.

### 12.3 CLAUDE.md와 codified-context.md의 관계

CLAUDE.md = "이 프로젝트가 뭔지" (what). codified-context.md = "context 시스템을 어떻게 쓰는지" (how). 둘을 합쳐서 논문의 Constitution(Tier 1)에 대응.

### 12.4 Subdirectory CLAUDE.md와 _overview.md 역할 분리

`src/auth/CLAUDE.md` (Claude Code 자동 로드) = "이 디렉토리에서 코드 쓸 때의 규칙" (convention).
`context/specs/auth/_overview.md` (탐색 후 로드) = "이 모듈이 어떻게 동작하는지" (spec).

용도가 다르므로 공존 가능. Subdirectory CLAUDE.md에서 overview를 참조하도록 연결:

```markdown
# src/auth/CLAUDE.md
이 모듈의 상세 spec: context/specs/auth/_overview.md 참조
관련 decisions: context/decisions/001-chose-local-auth.md 참조
```

## 14. Git worktree 전략

### 14.1 Worktree에서의 파일 가용성

| 파일 | Tracked? | Worktree에서 자동 가용? |
|------|----------|----------------------|
| `CLAUDE.md` | Yes | ✅ 자동 |

| `tools/drift-warning.js` | Yes | ✅ 자동 |

### 14.2 Context 문서 conflict 전략

**Frontmatter 자동 필드 conflict:** `last_commit`, `last_synced`를 두 worktree에서 각각 업데이트하면 conflict. → 아무 쪽이든 accept 후 `cc-drift-check` 실행. 자동 재계산됨.

**Conflict 예방 원칙:**
- Spec은 가능한 작은 단위(component level)로 쪼갠다.
- ADR/roadmap은 append-only(새 파일 추가)를 기본으로 한다.
- Conventions는 섹션 단위 분리를 권장.


### 15.2 codified-context.md의 섹션 분류

`codified-context.md`는 tool-managed 섹션과 project-specific 섹션이 공존한다. cc-upgrade는 섹션 단위로 merge한다.

| 섹션 | 분류 | cc-upgrade 시 |
|------|------|--------------|
| Pre-work, Post-work, Knowledge capture 등 | TOOL-MANAGED | 교체 |
| Action routing (기본 테이블) | TOOL-MANAGED | 교체 |
| Hallucination 방지, /compact 시 | TOOL-MANAGED | 교체 |
| Agent routing trigger table | PROJECT-SPECIFIC | 보존 |
| 기타 프로젝트별 커스텀 섹션 | PROJECT-SPECIFIC | 보존 |

### 15.3 cc-upgrade workflow

```
cc-upgrade
  1. 현재 tool version 확인
  2. GitHub repo에서 최신 버전 fetch
  3. 백업 생성 (rollback용)
  4. Section-based merge:
     a. TOOL-MANAGED 섹션: 새 버전으로 교체
     b. PROJECT-SPECIFIC 섹션: 보존 (extract → reattach)
  5. Diff preview 표시 (변경될 부분 + 보존될 부분)
  6. 개발자 승인 후 적용
  7. 새 agent 템플릿 → "추가할까요?" 확인
  8. Breaking change → migration guide 출력
  9. Project layer 절대 보존
```

## 16. Trigger table (optional, advanced)

논문의 trigger table은 파일 패턴 → 전문 에이전트 매핑. v2에서는 optional advanced feature.

- 작은 프로젝트 (custom agent 0~2개): 불필요
- 큰 프로젝트 (custom agent 5+개): 유용

`cc-add-agent` 시 "어떤 파일 영역을 담당하나요?" 질문 → trigger table에 행 추가. cc-onboard가 자동 생성하는 게 아니라 점진적으로 축적되는 구조.

```markdown
### Agent routing (codified-context.md 하단)
| Pattern | Agent | Timing |
|---------|-------|--------|
| src/payment/** | payment-specialist | pre-change |
| src/auth/** | auth-reviewer | post-change |
```

## Appendix A: Design decisions log

| # | 결정 | 이유 |
|---|------|------|
| D1 | 초기 버전은 MCP server 없이 CLI + skill/hook | adoption friction 최소화. 200+ 문서 시 MCP 도입 가능. |
| D2 | Constitution = CLAUDE.md + rules/codified-context.md | 별도 파일 불필요. 기존 메커니즘 활용. |
| D3 | CLAUDE.md는 개발자 소유, tool은 제안만 | 소유권 명확화. |
| D4 | covers 필드는 optional, convention-based 매핑 우선 | covers 자체가 drift 원인이 될 수 있음. |
| D5 | Symptom-Cause-Fix를 issue/agent spec 표준 포맷으로 | 논문의 실증적 효과. |
| D6 | cc-capture-lesson skill 추가 | 세션 내 knowledge capture 회고. |
| D7 | Codex 지원은 AGENTS.md indirection | 최소 비용으로 cross-tool 호환. |
| D8 | 새 프로젝트는 인터뷰 우선 onboarding | 분석할 코드가 없으므로. |
| D9 | confidence: intent 추가 | 새 프로젝트에서 "아직 구현 안 됨" 표현 필요. |
| D10 | Roadmap에 invalidates 대신 related_specs | 미래 예측의 strong claim은 위험. Soft warning으로 전환. |
| D11 | depends_on은 module overview에서만 | 유지보수 범위 폭발 방지. Tags 기반 검색으로 보완. |
| D12 | Frontmatter 자동 필드 (last_commit, last_synced) | 사람의 유지보수 부담 최소화. |
| D13 | .claude/rules/에 시스템 instruction 배치 | CLAUDE.md 소유권 충돌 방지. |
| D14 | Hook으로 instruction 이중 방어 | Instruction following ≠ 100%. |
| D15 | Rules file 중심 아키텍처 (v1의 Skill 중심에서 전환) | AI-driven maintenance. 개발자는 승인만. |
| D16 | Trigger table은 optional, cc-add-agent 시 점진 축적 | 범용 도구에서 모든 프로젝트가 domain agent를 갖진 않음. |
| D17 | Knowledge capture는 instruction + cc-capture-lesson 이중 구조 | 실시간 + 회고 보완 관계. |
| D18 | Spec 계층적 디렉토리 구조 | Worktree conflict 분산 + zoom level 지원 + 대규모 프로젝트 대응. |
| D19 | Onboarding Phase 2를 blocking 설계 | project.md, ADR, roadmap은 코드에 없는 정보. 인터뷰 필수. |
| D20 | suggest_context/search_context 분리 유지 | "파일 기반 조회"와 "개념 기반 탐색"은 의도가 다름. |
| D21 | verification-agent를 persistent agent로 승격 | self-verification은 약함. 독립 context window에서 검증해야 hallucination 방지 효과. spec 생성/수정 시마다 즉시 실행. |
| D22 | context-gardener에 pruning 역할 추가 | dead spec(소스 삭제됨), resolved issue, completed roadmap이 축적되면 noise 증가 → AI 품질 저하. |
| D23 | Skill 기반 Action routing + 명시적 테이블 | Rules file(WHEN) → Skill(HOW) → Agent(WHO) 역할 분리. 암시적 매칭보다 deterministic. |
| D24 | Onboarding Phase 2를 자동 초안 생성으로 변경 | git/코드 분석으로 decisions, issues, project.md 초안 자동 생성. 인터뷰는 검증+보완으로 경량화. 개발자 시간 절약. |
| D25 | last_commit 필드 제거, last_synced만 유지 | commit hash는 커밋 후에야 알 수 있어 닭-달걀 문제 발생. date 기반이 관리가 단순하고 drift detection에 충분. |
| D26 | Source roots 설정 추가 | convention-based 매핑이 src/ 이외 구조, 모노레포에서 깨지는 문제 해결. cc-onboard 시 자동 감지. |
| D27 | Glossary는 링킹으로 접근 | 별도 로드 메커니즘 없이 CLAUDE.md/spec에서 링크. 이미 로드되는 문서의 링크를 따라 자연스럽게 접근. |
| D28 | cc-upgrade는 section-based merge | sprint update-version 패턴 적용. TOOL-MANAGED/PROJECT-SPECIFIC 섹션 구분하여 커스텀 내용 보존. |
| D29 | Hook 구현 언어: Node.js/Python | bash로 YAML frontmatter 파싱은 깨지기 쉬움. 대부분 프로젝트에 런타임이 이미 존재. |
